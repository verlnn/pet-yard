# PetYard 프론트엔드 디자인 가이드

이 문서는 새 Codex 세션에서도 PetYard의 프론트엔드 디자인 언어가 흔들리지 않도록 현재 시각적 기준을 정리한 문서입니다.

## 1. 기본 방향
- 톤: 따뜻하고, 차분하고, 친근하며, 과하지 않게 정돈된 느낌
- 기본 모드: 라이트 우선
- 시각 분위기: 부드러운 그라데이션, 큰 라운드, 은은한 카드 깊이, 가벼운 유리감
- 피해야 할 것: 거친 다크 UI, 네온 포인트, 지나치게 평평한 레이아웃, 페이지마다 달라지는 스타일

## 2. 색상 시스템
기본 팔레트는 현재 Tailwind 확장 색을 기준으로 사용한다.

- `ink`: 기본 텍스트, 주요 버튼 배경, 강한 강조
- `sand`: 앱 기본 배경
- `pine`: 보조 강조
- `moss`: 자연스러운 보조 포인트
- `clay`: 따뜻한 강조/배경 포인트
- `ember`: 위험/경고/삭제
- `sky`: 차가운 배경 포인트

규칙:
- 기본 대비는 `ink + sand` 조합을 우선한다.
- `pine`, `moss`, `clay`, `sky`는 보조 포인트로 사용하고 주색으로 남용하지 않는다.
- `ember`는 에러, 경고, 파괴적 액션에만 제한적으로 사용한다.
- 특별한 이유가 없으면 새로운 메인 브랜드 색을 만들지 않는다.

## 3. 타이포그래피
- 디스플레이 폰트: `Space Grotesk`
- 본문 폰트: `SUIT`

규칙:
- 제목과 섹션 헤더는 display 계열을 사용한다.
- 본문, 라벨, 폼 텍스트, 도움말은 body 계열을 유지한다.
- 장식보다 정보 위계가 먼저 보이게 한다.

## 4. 배경과 Surface
- 기본 페이지 배경은 완전히 평평하기보다 레이어감이 느껴져야 한다.
- 기존 배경은 부드러운 radial gradient와 따뜻한 중성톤을 사용해 왔다.
- 카드는 보통 아래 성격을 가진다.
  - 반투명 흰색
  - 은은한 border
  - 필요한 경우 backdrop blur
  - 부드러운 그림자

규칙:
- “따뜻한 배경 위 부드러운 카드” 방식을 우선 재사용한다.
- 중요한 컨테이너는 무거운 단색보다 은은한 반투명 표현을 우선한다.
- 미디어 뷰어처럼 이미 어두운 콘텍스트가 아닌 이상 과도하게 어두운 패널은 피한다.

## 5. 라운드와 형태
- 인터페이스 전반에서 큰 라운드를 일관되게 사용한다.
- 자주 쓰는 패턴
  - 입력창/버튼: `rounded-2xl`
  - pill/tag: `rounded-full`
  - 큰 패널/다이얼로그: `28px` ~ `32px`

규칙:
- 기본적으로 넉넉한 라운드를 사용한다.
- 전체 시스템이 큰 라운드인데 일부만 각진 형태로 섞지 않는다.

## 6. 그림자와 깊이
- 그림자는 부드럽고 살짝 떠 있는 느낌이어야 한다.
- 기존 패턴은 저대비의 넓은 blur shadow를 사용한다.

규칙:
- 과한 elevation보다 은은한 깊이를 선호한다.
- hover 모션은 가볍게 떠오르거나 배경이 바뀌는 정도로 유지한다.
- 강한 scale, 과한 drop shadow는 피한다.

## 7. 입력창과 폼 컨트롤
현재 공통 스타일 방향:
- 큰 라운드 입력창
- 부드러운 border
- 흰색 또는 거의 흰색 배경
- 은은한 shadow
- `ink` 계열 focus ring

규칙:
- 새 폼도 기존 auth/onboarding 계열 입력창과 같은 가족으로 맞춘다.
- 새 변형을 만들기 전에 기존 패턴을 먼저 확장할 수 있는지 확인한다.
- disabled 상태도 흐리되 읽을 수 있어야 한다.

## 8. 버튼
현재 위계:
- Primary: 짙은 `ink` 배경 + `sand` 텍스트
- Secondary/Ghost: 흰색 또는 반투명 흰색 + border
- Destructive: 필요한 경우만 red/ember 계열

규칙:
- 한 화면이나 한 섹션에는 명확한 primary action 하나를 둔다.
- 온보딩 같은 step 기반 흐름에서는 full-width primary 버튼을 적극적으로 사용한다.
- 보조 버튼은 primary와 같은 무게를 가지지 않도록 한다.

## 9. 모션
- 모션은 존재하지만 절제되어 있어야 한다.
- 현재 스타일은 fade/slide와 가벼운 hover lift를 선호한다.

규칙:
- 모션은 장식이 아니라 흐름을 설명하는 데 사용한다.
- 선호 패턴:
  - fade up
  - subtle translate
  - soft opacity transition
- 과한 바운스, flashy animation, 일관성 없는 곡선은 피한다.

## 10. 컴포넌트 네이밍과 스타일링
- JSX 안에는 역할이 명확한 경우 긴 utility 나열보다 semantic class name을 우선한다.
- 구조적으로 의미 있는 스타일은 `web/app/globals.css`의 `@layer components`로 모은다.

예시:
- `onboarding-profile-step-panel`
- `onboarding-pet-verify-button`
- `feed-detail-menu-trigger`

## 11. 새 작업의 일관성 규칙
새 UI를 만들거나 수정할 때는 아래 순서를 따른다.

1. 기존 컴포넌트나 스타일 패턴이 이미 같은 문제를 해결하고 있는지 확인한다.
2. 있다면 새 스타일을 만들기 전에 기존 패턴을 확장한다.
3. 정말 새 패턴이 필요하다면, 기존 시스템의 자연스러운 구성원처럼 보이게 만든다.

## 12. 피해야 할 것
- 페이지마다 랜덤하게 달라지는 디자인
- 화면마다 서로 다른 버튼 철학
- 현재 팔레트를 압도하는 새로운 색군
- 큰 라운드 시스템 안에서 갑자기 작고 답답한 radius
- semantic class로 풀 수 있는데 JSX에 utility만 과도하게 나열하는 방식

## 13. 다크 플로팅 패널 시스템
이제 PetYard에는 사이드바 `더보기` 메뉴, `모드 전환` 패널, `로그아웃 확인` 같은 alert 카드에 공통으로 쓰는 다크 플로팅 패널 패턴이 있다.

이 패턴은 일회성 스타일이 아니라 재사용 가능한 하나의 UI 계열로 취급해야 한다.

이 패턴을 써야 하는 경우:
- `더보기` 같은 컨텍스트 메뉴
- 가벼운 설정 서브패널
- 로그아웃처럼 짧은 확인 액션 카드
- 제품 내부에서 같은 계열로 느껴져야 하는 alert/confirm 오버레이

이 패턴을 쓰지 말아야 하는 경우:
- 전체 화면 다이얼로그
- 폼이 많은 무거운 모달
- 긴 콘텐츠를 담는 대형 팝업
- 온보딩 step 화면

### 13.1 시각적 정체성
이 패널 계열은 아래 인상을 가져야 한다.
- 어둡다
- 컴팩트하다
- 떠 있다
- 둥글다
- 과시적이지 않고 정돈돼 있다

기준 레퍼런스는 현재 사이드바 `더보기` 패널이다.
- 짙은 차콜 계열 배경
- 큰 라운드 코너
- 깊지만 부드러운 그림자
- 얇은 divider
- 흰색 중심 텍스트와 살짝 눌린 보조 텍스트

이 계열은 기본 라이트 카드 시스템과는 다르다. 기본 카드의 대체제가 아니라, 오버레이 전용으로 쓰는 특수 계열이다.

### 13.2 Surface 규칙
권장 표면 특성:
- 임의의 검정보다 공통 menu surface token을 사용한다
- 코너 라운드는 대략 `24px` 수준으로 유지한다
- overflow는 잘려야 한다
- 그림자는 강하되 번짐이 부드러워야 한다
- 내부 여백은 넉넉하되 과하지 않게 유지한다

규칙:
- 패널 안에 작은 카드 여러 장을 쌓기보다 하나의 연속된 surface로 보여야 한다.
- 떠 있는 오브젝트 하나처럼 읽혀야 한다.
- divider는 얇고 은은해야 한다.
- 밝은 외곽선, glow, 장식용 그라데이션은 넣지 않는다.

### 13.3 정보 위계
이 패널 계열에서는 정보 위계가 즉시 보여야 한다.
- 제목이나 핵심 라벨이 가장 먼저 보여야 한다
- 보조 설명은 톤을 낮추되 읽을 수 있어야 한다
- 액션은 메시지보다 더 강해 보이면 안 된다

특히 alert 카드에서는:
- 문구가 버튼보다 더 가시적이어야 한다
- 버튼이 제목과 경쟁하면 안 된다
- 큰 CTA 버튼보다 컴팩트한 액션 row가 더 적합하다

### 13.4 액션 영역 규칙
이 다크 패널 계열의 액션은 일반적인 “모달 버튼”보다 “메뉴 row”에 가까운 느낌이어야 한다.

규칙:
- 액션은 컴팩트하게 유지한다
- 버튼 텍스트는 `text-sm` 또는 그보다 약간 작은 수준을 우선한다
- 과하게 큰 패딩 버튼은 피한다
- 액션이 여러 개면 divider를 사용한다
- 짧은 액션 2개만 있을 때는 좌우 배치도 허용한다

좌우 배치일 때:
- 레이블은 가운데 정렬한다
- 두 액션은 시각적으로 균형을 맞춘다
- 사이에는 세로 divider를 둔다

상하 배치일 때:
- 레이블은 좌측 정렬을 우선한다
- 메뉴 리스트 같은 인상을 유지한다
- 행 사이에는 가로 divider를 둔다

### 13.5 모드 전환 패널 패턴
`모드 전환` 서브패널은 `더보기` 메뉴와 완전히 같은 카드 계열이어야 한다.
- 같은 어두운 surface
- 같은 radius
- 같은 shadow 언어
- 같은 divider 처리

기본 구조:
1. 헤더 row
2. 은은한 divider
3. 컴팩트한 설정 row

헤더 row 규칙:
- 좌측에는 back control
- 중앙 또는 시각적으로 우세한 위치에 제목
- 우측에는 현재 모드 상태를 보여주는 아이콘

아이콘 규칙:
- 다크모드가 켜져 있으면 달 아이콘을 사용한다
- 라이트모드가 켜져 있으면 해 아이콘을 사용한다
- 아이콘은 “바꾸면 어디로 가는지”보다 “현재 상태가 무엇인지”를 먼저 보여줘야 한다

### 13.6 Alert 카드 패턴
alert 성격의 카드는 이제 ad hoc 마크업으로 만들지 않고 공용 컴포넌트로 구현한다.

현재 공통 방향:
- 공용 alert 컴포넌트를 사용한다
- 렌더링은 viewport 기준 overlay에 한다
- 카드 외형은 `더보기` 메뉴 패널과 같은 가족으로 유지한다

기본 구조:
1. 강한 제목
2. 짧은 설명
3. divider
4. 액션 그룹

동작 규칙:
- 적절한 경우 backdrop 클릭으로 닫을 수 있어야 한다
- `Escape`로 닫을 수 있어야 한다
- 카드는 화면 가운데에 떠야 한다
- transform이 걸린 부모 레이아웃의 영향을 받으면 안 된다

### 13.7 권장 공용 API
재사용 가능한 alert UI는 아래 속성을 받을 수 있어야 한다.
- title
- description
- confirm label
- cancel label
- action layout class 또는 variant
- close handler
- confirm handler

액션 레이아웃은 설정 가능해야 한다. 이유는 alert마다 더 자연스러운 배치가 다르기 때문이다.
- 좌우형 compact action
- 상하형 stacked action

이 패턴이 확장될 때도 inline utility 나열보다 semantic class family를 우선한다.

### 13.8 구현 절차
이 패턴을 구현할 때는 아래 순서를 따른다.

1. 기존 `더보기` 메뉴 스타일을 기준으로 시작한다.
2. 같은 surface token과 divider 언어를 재사용한다.
3. 간격은 컴팩트하게 유지한다.
4. 액션보다 문구가 먼저 보이도록 만든다.
5. alert 성격의 UI는 반드시 공용 컴포넌트로 만든다.
6. 기존 패턴으로 해결 가능한데 새 변형을 섣불리 만들지 않는다.
