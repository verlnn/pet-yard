# Codex Front End 작업 규칙

이 문서는 이 저장소의 프론트엔드 작업에서 Codex가 따라야 할 기본 규칙을 정리한 한글 안내 문서입니다.

## 기본 응답 방식
- 기본적으로 한국어로 응답한다.
- 설명은 짧고 실무적으로 한다.

## 커밋 규칙
- 사용자가 구현 작업을 요청했고, 별도로 커밋하지 말라고 하지 않았다면 작업 완료 후 자동으로 커밋한다.
- 커밋 메시지는 한글로 작성한다.
- 사용자가 한 번에 서로 다른 기능이나 변경사항을 여러 개 요청했다면, 가능하면 기능 단위로 나눠서 각각 별도 커밋한다.
- 가능하면 아래 형식을 사용한다.
  - `type(scope): 설명`
- 예시
  - `feat(onboarding): 반려동물 온보딩 등록번호 인증 흐름 추가`
  - `refactor(onboarding): 프로필 온보딩 페이지를 단계별 컴포넌트로 분리`
  - `fix(ui): 온보딩 버튼 비활성화 조건 정리`

## 클래스 네이밍 규칙
- JSX 안에 의미 없는 긴 Tailwind utility 문자열을 그대로 남기지 않는다.
- 역할이 드러나는 semantic class name을 우선 사용한다.
- 예시
  - `onboarding-profile-step-panel`
  - `onboarding-profile-primary-button`
  - `feed-detail-menu-trigger`
- 실제 스타일 정의는 가능하면 `web/app/globals.css`의 `@layer components`로 옮긴다.
- 기존 프론트 파일을 수정할 때 class naming이 읽기 어렵거나 utility 나열 위주라면, 가능한 범위에서 semantic class 중심으로 같이 정리한다.

## 디자인 일관성 규칙
- 시각적 기준은 `web/docs/DESIGN_GUIDE.md`를 따른다.
- 다크모드 작업이 포함되면 `web/docs/DARK_MODE_GUIDE.md`도 함께 따른다.
- 새 세션마다 다른 스타일을 만들지 말고, 현재 PetYard 디자인 언어와 최대한 일관되게 작업한다.
- 특별한 이유가 없으면 기존 색상 팔레트, radius, shadow, 입력창 스타일, 버튼 위계를 재사용한다.
- 새로운 화면을 만들 때도 먼저 현재 디자인 시스템에 맞는 형태로 풀 수 있는지 확인한다.

## 컴포넌트 분리 규칙
- 하나의 프론트 컴포넌트가 너무 많은 역할을 가지면 먼저 분리한다.
- 역할이나 기능 기준으로 분리하고, 임의로 쪼개지 않는다.
- 페이지 컴포넌트는 가능한 한 아래 역할만 담당한다.
  - 라우팅
  - API 호출
  - 상태 orchestration
  - 단계 흐름 제어
- 렌더링이 큰 UI 블록은 별도 컴포넌트로 분리한다.
- 우선적으로 분리할 대상 예시
  - 진행바
  - 단계별 패널
  - 재사용 필드 그룹
  - 모달/메뉴/상세 패널

## 프론트 수정 시 우선 확인할 것
프론트 코드를 수정할 때는 먼저 아래를 본다.

1. class name이 의미 있게 작성되어 있는가
2. 한 컴포넌트가 너무 많은 기능과 책임을 가지고 있지 않은가

둘 중 하나라도 아니면, 구조를 먼저 정리한 뒤 요청 작업을 이어서 진행한다.

## 검증 규칙
- 프론트 변경 후 가능하면 `npx tsc --noEmit`를 실행한다.
- 화면 구조 변경이 크면 빌드 오류가 없는지도 같이 확인한다.
- 검증을 못 했으면 그 사실을 명시한다.

## 협업 원칙
- 새 세션에서 사용자가 같은 프론트엔드 작업 원칙을 다시 설명하지 않아도 되도록 이 문서를 기본 기준으로 사용한다.
- `web/` 아래 프론트 작업은 이 문서를 우선 기준으로 따른다.
