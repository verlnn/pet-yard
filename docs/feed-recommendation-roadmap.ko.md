# 홈 피드 추천 확장 로드맵

## 현재 상태
- 현재 홈 피드는 최신순 홈 피드다.
- cursor는 `createdAt desc, id desc` 기준으로 동작한다.
- 지금의 cursor 계약은 타임라인 탐색에는 적합하지만, 추천/랭킹 혼합 피드에는 최적화되어 있지 않다.

## 이번 변경 내용
- 추천 피드를 실제로 구현하지는 않았다.
- 대신 현재 최신순 계약의 유효 범위와, 앞으로 추천 피드로 갈 때 어떤 방향이 가능한지 문서로 명확히 정리했다.

## 현재 홈 피드의 성격
- 정렬 규칙이 결정적이다.
- `createdAt + id`로 이어보기가 안정적이다.
- 무한 스크롤 동작을 예측하기 쉽다.
- 점수 기반 피드보다 디버깅이 단순하다.

그래서 현재 구조는 아래 성격에 잘 맞는다.
- 최신순 탐색
- 안정적인 cursor pagination
- `maxPages` 기반의 제한된 클라이언트 캐시

## 추천 확장 시나리오

### 1. 홈은 최신순 유지, 추천은 별도 surface로 분리
- 장점
  - 위험이 가장 낮다
  - 현재 홈 계약을 그대로 유지할 수 있다
  - A/B rollout이 쉽다
- 단점
  - 사용자가 surface를 전환해야 한다
  - engagement가 분산될 수 있다
- 현재 구조와의 궁합
  - 가장 좋다
  - 첫 단계로 가장 권장되는 방식이다

### 2. 홈에서 최신순 게시물과 추천 게시물을 혼합
- 장점
  - 추천을 메인 홈 안에 자연스럽게 노출할 수 있다
  - 점진적 rollout이 가능하다
- 단점
  - 정렬 규칙이 복잡해진다
  - 디버깅 난도가 올라간다
  - 광고와 추천이 같은 인벤토리를 두고 경쟁하게 된다
- 현재 구조와의 궁합
  - 가능은 하지만, mixed item 계약과 cursor 의미를 더 엄격하게 재정의해야 한다

### 3. 점수(score) 기반 단일 홈 피드
- 장점
  - 개인화 유연성이 가장 크다
  - 하나의 랭킹 시스템으로 전체 피드를 제어할 수 있다
- 단점
  - 구현 비용이 가장 크다
  - 최신순 기대가 약해진다
  - 캐시 무효화와 디버깅이 훨씬 어려워진다
- 현재 구조와의 궁합
  - 전용 feed-serving 계층 없이 바로 가기에는 가장 부담이 크다

## cursor가 앞으로 달라질 수 있는 방식

### score + id cursor
- 예시 정렬:
  - `score desc, id desc`
- 랭킹이 score 중심일 때 적합하다.
- 위험:
  - cursor 발급 후 점수가 바뀌면 순서가 달라질 수 있다.

### publish time + score 혼합 cursor
- 예시 정렬:
  - `publish_bucket desc, score desc, id desc`
- “신선하지만 관련도도 있는” 피드에 적합하다.
- 위험:
  - 정렬 의미가 복잡해져 설명과 테스트가 어려워진다.

### 서버 발급 feed token
- 서버가 랭킹 스냅샷을 나타내는 opaque token을 내려준다.
- 클라이언트는 다음 요청에서 그 token을 다시 보낸다.
- 랭킹이 동적이거나 실험이 많을 때 유리하다.
- 위험:
  - 더 상태적인 feed-serving 전략이 필요하다.

## 추천 신호 후보
- 좋아요
- 댓글
- 체류 시간
- 숨김 / 신고 / 차단
- 팔로우 그래프
- 사용자 관심사
- 반려동물 관심사
- 지역 친화도
- 최신성 감쇠

## 지켜야 할 기준
- 새 계약이 문서화되기 전까지 현재 홈 피드는 최신순 피드라는 점을 명확히 유지할 것
- 테스트와 문서 수정 없이 기존 cursor에 추천 의미를 섞지 말 것
- 광고와 추천은 서빙 계층이 둘 다 통제하기 전까지 개념적으로 분리할 것

## 남아 있는 한계
- 아직 추천 API는 없다.
- 랭킹 저장소나 scoring pipeline도 없다.
- 추천 item 계약도 아직 없다.

## 추후 개선 포인트
- 먼저 별도 추천 탭 또는 API부터 추가
- 광고 계약과 나란히 추천 item 계약 정의
- 랭킹이 비결정적으로 변하기 시작하면 token 기반 continuation 검토
