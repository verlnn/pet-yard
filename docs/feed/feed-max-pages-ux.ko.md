# 피드 maxPages UX

## 현재 상태

홈 피드는 React Query 캐시에 유지하는 페이지 수를 의도적으로 제한하고 있습니다.

현재 정책:

- `MAX_HOME_FEED_PAGES_IN_MEMORY = 6`

이 값을 두는 이유:

- 장시간 세션에서 메모리가 무한히 증가하지 않도록 하기 위해
- 이미지가 많은 피드 카드가 무한정 React tree에 남지 않게 하기 위해
- 피드 캐시 압력을 일정 수준 이하로 유지하기 위해

## 트레이드오프

`maxPages`를 쓰면 사용자가 계속 스크롤할 때 오래된 페이지는 메모리에서 제거될 수 있습니다.

현재 단계에서는 합리적인 선택이지만, UX 측면에서 아래 영향이 있습니다.

- 아주 깊게 내려간 뒤 다시 돌아오면 재요청이 생길 수 있음
- 스크롤 위치 복원이 없으면 재진입 체감이 끊길 수 있음
- stale cache 구간과 토큰 변경 상황에서 사용자가 보는 데이터가 달라질 수 있음

## 이번 변경 내용

### 스크롤 상태 저장/복원

홈 피드는 이제 사용자별 scroll state를 session storage에 저장합니다.

현재 동작:

- 스크롤 중 `window.scrollY` 저장
- unmount 시 한 번 더 저장
- 피드 재진입 후 첫 페이지가 준비되면 한 번만 복원

키는 access token 기준으로 분리되므로, 다른 사용자로 전환했을 때 이전 위치가 섞이지 않습니다.

### query key 분리와 정합성

이미 query key는 토큰 기준으로 분리되어 있었고, 이번에 scroll state도 같은 기준으로 분리했습니다.

그래서 아래 상황에서 정합성이 좋아졌습니다.

- 계정 전환
- 로그아웃/로그인
- 사용자 간 캐시 재사용 방지

## 실제 UX 시나리오

### 오래 스크롤한 세션

- 오래된 페이지는 메모리에서 제거될 수 있음
- 현재 보이는 페이지는 그대로 사용 가능
- 나중에 복귀하면 제거된 페이지는 재요청될 수 있음

### 중간 게시물 진입 후 복귀

현재 홈 피드에는 독립된 상세 라우트가 없지만, 피드 페이지 자체로 되돌아오는 경우 scroll restore 덕분에 체감 끊김이 줄어듭니다.

### 탭 전환 / 잠시 이탈 후 재진입

- 캐시 창이 남아 있으면 React Query가 재사용
- 일부 페이지가 제거됐더라도 scroll restore가 위치 회복을 도와줌

### 토큰 변경

- feed query cache는 토큰별 분리
- scroll state도 토큰별 분리
- 사용자 간 캐시 오염 방지

## 언제 정책을 다시 검토해야 하는가

현재 전략은 아래 조건에서는 여전히 유효합니다.

- 카드 높이가 가변적임
- 피드가 주로 최신순 타임라인임
- 세션 길이가 아직 극단적으로 길지 않음
- 현재 `maxPages` 값이 눈에 띄는 불편을 만들지 않음

아래 시점에는 더 강한 windowing / virtualization 재검토가 필요합니다.

- 평균 세션 depth가 크게 증가
- 저사양 기기에서 메모리 사용량이 실제 문제로 드러남
- 재진입/뒤로가기 시 재요청 비용이 자주 불만으로 이어짐
- 피드 아이템 종류가 훨씬 더 다양해짐

## 남아 있는 한계

- scroll restore는 현재 `scrollY` 기준이지 item anchor 기준은 아님
- 제거된 페이지는 여전히 재요청이 필요할 수 있음
- 홈 피드 전용 상세 라우트 상태까지 보존하는 구조는 아직 아님

## 다음 개선 포인트

후속 후보:

1. 단순 `scrollY`가 아니라 item id anchor 기반 복원
2. 최근에 본 피드 구간에 대한 작은 back buffer 유지
3. 독립된 피드 상세 라우트가 생기면 route-aware 복귀 UX 도입
